# 大模型框架/平台对比与适用场景

本文对比了以下大模型相关框架和平台，并给出了各自适合的应用场景：

- Ollama
- vLLM
- SGLang
- FastChat
- LLaMA Factory
- One-API
- LocalAI

---

## 1. 框架概览

| 框架/平台               | 类型                 | 特点                                               | 适合场景                           | 优势                                      | 局限                                 |
| ----------------------- | -------------------- | -------------------------------------------------- | ---------------------------------- | ----------------------------------------- | ------------------------------------ |
| **Ollama**        | 本地 & 云端 LLM 管理 | 专注 Mac/Windows 本地运行大模型，提供简单 API 接口 | 个人开发者、原型验证、快速本地测试 | 轻量、易用，开箱即用                      | 模型生态相对有限，对 GPU 扩展有限    |
| **vLLM**          | 高性能推理库         | 针对大模型的高吞吐量推理优化                       | 多 GPU 分布式推理、高并发服务      | 高性能，低延迟，支持 tensor parallel      | 仅推理，不管理模型生命周期           |
| **SGLang**        | 多模型统一接口框架   | 提供统一接口对接不同模型                           | 企业级应用、嵌入式服务             | 多模型兼容性强，便于迁移                  | 社区较小，文档资源有限               |
| **FastChat**      | LLM 聊天系统框架     | 提供聊天机器人系统、消息管理、模型调度             | 构建本地或云端聊天系统             | 成熟的聊天系统实现，支持多模型切换        | 主要专注聊天，不通用推理或 embedding |
| **LLaMA Factory** | 模型管理与推理平台   | 对 LLaMA 系列模型做封装和管理                      | 模型训练/推理快速部署              | 易于管理 LLaMA 模型，提供统一接口         | 仅针对 LLaMA 系列                    |
| **One-API**       | 模型接口统一平台     | 类似 OpenAI API 的统一调用接口                     | 企业多模型服务，SaaS 应用          | 统一接口调用不同模型，支持 embedding/chat | 依赖 One-API 平台，可能有限制        |
| **LocalAI**       | 本地部署 LLM 服务器  | 本地部署 OpenAI 兼容接口，支持多种模型             | 本地推理、离线环境、隐私敏感场景   | 支持 OpenAI API 协议，方便替换            | 模型管理需自己处理，部署成本高       |

## 2. 深入对比概括

### 2.1 推理性能

- **vLLM** → 专注高性能推理，低延迟，高吞吐量，支持分布式。
- **FastChat / Ollama / LocalAI** → 中等性能，适合单机或小规模并发。
- **SGLang / One-API / LLaMA Factory** → 侧重接口和管理，不是推理优化工具。

### 2.2 模型管理

- **Ollama / LLaMA Factory** → 本地模型管理方便。
- **One-API / SGLang** → 多模型统一管理接口，适合企业。
- **vLLM** → 只关注推理，不管理模型。

### 2.3 部署场景

| 场景               | 推荐框架/平台   |
| ------------------ | --------------- |
| 个人本地测试       | Ollama, LocalAI |
| 企业多模型统一调用 | SGLang, One-API |
| 高并发聊天机器人   | FastChat + vLLM |
| LLaMA 系列专用部署 | LLaMA Factory   |
| 离线部署，隐私敏感 | LocalAI         |

### 2.4 功能对比（API / Chat / Embedding）

| ------------- | ---- | --------- | ---------- | ------------------ |

| SGLang        | ✅   | ✅        | ❌         | ❌                 |

## 3. 总结

1. **本地快速原型** → Ollama, LocalAI
2. **高并发推理** → vLLM
3. **聊天系统** → FastChat + vLLM
4. **企业级统一接口** → SGLang, One-API
5. **LLaMA 系列优化** → LLaMA Factory

## vllm调用时候模型资源

### 资源分析

| 组成部分                             | 占用说明                                         | 是否受 `gpu_memory_utilization` 控制 |
| ------------------------------------ | ------------------------------------------------ | -------------------------------------- |
| 1. 模型权重（Model Weights）         | 模型参数（float16/bfloat16），常驻显存           | ❌ 不受控制（必须加载）                |
| 2. KV Cache（PagedAttention 缓存）   | 存储每个 token 的 key/value，随 batch 和长度增长 | ✅ 受 `gpu_memory_utilization` 控制  |
| 3. 临时工作空间（Temporary Buffers） | Attention 计算、softmax、采样等中间结果          | ❌ 不受控制，动态分配                  |
| 4. 显存碎片 & 对齐开销               | CUDA 分配器需要对齐块，可能产生碎片              | ❌ 无法精确预测                        |
| 5. 系统/驱动/其他进程                | NVIDIA 驱动、桌面 GUI、监控工具等                | ❌ 不可控                              |

### 实例说明

举个实际例子（以24G GPU为例）
假设：
    模型：Qwen-1.7B  约3.4GB
    设置： gpu_memory_utilization = 0.8
    总显存：24G

```
总显存：24 GB
│
├── 模型权重：3.4 GB（固定）
├── KV Cache 最多：24 × 0.8 = 19.2 GB → 实际可用 ≈ 19.2 - 3.4 = 15.8 GB？
└── 剩余 20%（4.8 GB）：留给临时计算 + 安全边际
```

### vllm参数解析

- batch_size

  - '喂饭的速度’
- `max_num_seqs`

  - vllm内部的消化能力
- `gpu_memory_utilization`

  - 值越大 → 能缓存更多请求的中间状态 → 支持更长上下文或更高并发
  - 值太小 → 浪费显存，降低吞吐
  - 值太大 → 可能 OOM（Out-Of-Memory）

| 参数                              | 含义                           | 控制内容                                                | 作用对象     | 是否包含 Prompt    | 影响资源                    | 典型值           |
| --------------------------------- | ------------------------------ | ------------------------------------------------------- | ------------ | ------------------ | --------------------------- | ---------------- |
| `max_seq_len（其他框架，非LLM)` | 最大序列长度（上下文窗口上限） | 单条请求中**prompt + 生成内容** 的总 token 数上限 | 单条请求     | ✅ 是              | 显存（KV Cache 随长度增长） | 2048, 4096, 8192 |
| `max_tokens`                    | 最大生成 token 数              | 单条请求中**最多生成的新 token 数**               | 单条请求     | ❌ 否（仅输出）    | 显存 + 推理时长             | 128, 256, 512    |
| `max_num_seqs(VLLM中)`          | 最大并发序列数                 | 一个 batch 中**最多同时处理的请求数量**           | 整个推理批次 | —（每条独立计算） | 总显存 + 吞吐量             | 16, 32, 64, 256  |

| 参数                       | 含义                                       | 作用阶段                  | 典型值                          | 备注                                                       |
| -------------------------- | ------------------------------------------ | ------------------------- | ------------------------------- | ---------------------------------------------------------- |
| `max_model_len`          | 模型支持的最大上下文长度（input + output） | 限制单个请求的最大长度    | `4096`,`32768`,`"auto"`   |                                                            |
| `max_num_batched_tokens` | 所有并发请求的总 token 数上限              | 控制 GPU 显存使用，防 OOM | `32768`,`262144`,`524288` | `max_num_batched_tokens ≈ max_num_seqs × max_model_len`` |

```python
    "骚扰-外呼频次": "freq_call",

ID_TO_LABEL = {

    "4": "骚扰-不再致电",
    "5": "态度",
    "6": "骚扰-外呼频次",
    "7": "投诉倾向",

def normalize_label(label: str) -> str:
    return "&".join(sorted(label.split("&")))

# key_map的英文 -> 中文的映射
eng2zh = {v: k for k, v in key_map.items()}

# ID_TO_LABEL 的英文 --> label -> id 反查
LABEL_TO_ID = {
    normalize_label(v): k
    for k, v in ID_TO_LABEL.items()
}

def pred_emotion_to_label(pred_emotion):
    """
    输入:
      - dict
      - 或 json 字符串
    输出:
      - (label_id, label_name)
    """
    if pd.isna(pred_emotion):
        return pd.Series(["0", "其他"])

    # 如果是字符串，先反序列化
    if isinstance(pred_emotion, str):
        try:
            pred_emotion = json.loads(pred_emotion)
        except (json.JSONDecodeError, TypeError) as e:
            # 解析失败：视为无命中，返回“其他”
            # 可选：记录日志或打印错误（调试用）
            print(f"JSON 解析失败: {e} | 原始输入: {repr(pred_emotion)}")
            pred_emotion = {"attitude": 0, "no_call": 0, "freq_call": 0, "complaint": 0}

    # 命中的中文标签
    hit_labels = [eng2zh[k] for k, v in pred_emotion.items() if v == 1]

    if not hit_labels:
        label_name = "其他"
    else:
        # 排序保证和 ID_TO_LABEL 完全一致
        label_name = normalize_label("&".join(hit_labels))

    pred_label = LABEL_TO_ID.get(label_name, "0")
    return pd.Series([pred_label, label_name])

def run_inference(
    model_path: str,
    input_excel_path: str,
    output_excel_path: str,
    system_prompt: str,
    batch_size: int = 32,
    dtype: str = "bfloat16",
    gpu_memory_utilization: float = 0.87,
    max_num_seqs: int = 32,
    temperature: float = 0.5,
    top_p: float = 0.9,
    repetition_penalty: float = 1.1,
    max_tokens: int = 2048,
    is_label: bool = True,
    sheet_name:str = ''
):
    """
    使用 vLLM 对 Excel 中的文本进行批量情绪分类推理。

    Args:
        model_path: 模型路径（合并 LoRA 后的 checkpoint）
        input_excel_path: 输入 Excel 文件路径（需包含 'text_a' 列）
        output_excel_path: 输出 Excel 文件路径（会新增 'pred_emotion' 列）
        system_prompt: 系统提示词（情绪分类规则）
        batch_size: 推理批大小
        dtype: 模型精度（如 'bfloat16', 'float16'）
        gpu_memory_utilization: GPU 显存利用率
        max_num_seqs: vLLM 最大并发序列数
        temperature / top_p / repetition_penalty / max_tokens: 采样参数
    """
    logger.info("🔧 正在加载 tokenizer 和模型...")
    tokenizer = AutoTokenizer.from_pretrained(model_path, trust_remote_code=True)
    llm = LLM(
        model=model_path,
        dtype=dtype,
        gpu_memory_utilization=gpu_memory_utilization,
        trust_remote_code=True,
        max_num_seqs=max_num_seqs,
    )

    logger.info("📥 正在读取输入数据...")
    if not sheet_name :
        sheet_name = 0
    df = pd.read_excel(input_excel_path, sheet_name=sheet_name)
    # df = df.head()
    logger.info("📝 正在构建 prompts...")
    prompts = []
    for _, row in df.iterrows():
        user_query = str(row["text_a"]) if pd.notna(row["text_a"]) else ""
        messages = [
            {"role": "system", "content": system_prompt.strip()},
            {"role": "user", "content": user_query},
        ]
        prompt = tokenizer.apply_chat_template(
            messages, tokenize=False, add_generation_prompt=True
        )
        prompts.append(prompt)

    sampling_params = SamplingParams(
        temperature=temperature,
        top_p=top_p,
        repetition_penalty=repetition_penalty,
        max_tokens=max_tokens,
    )

    logger.info(f"🚀 开始批量推理（batch_size={batch_size}）...")
    torch.cuda.synchronize() if torch.cuda.is_available() else None
    start = time.time()

    results = []
    for i in tqdm(range(0, len(prompts), batch_size)):
        batch_prompts = prompts[i : i + batch_size]
        outputs = llm.generate(batch_prompts, sampling_params=sampling_params)
        for out in outputs:
            text = out.outputs[0].text.strip()
            # 移除 <think>...</think> 标签（如果模型输出中有）
            cleaned_text = re.sub(r"<think>.*?</think>", "", text, flags=re.DOTALL)
            results.append(cleaned_text.strip())

    torch.cuda.synchronize() if torch.cuda.is_available() else None
    end = time.time()
    if is_label:
        df["pred_label"] = results
    else:
        df["pred_emotion"] = results

    # 确保输出目录存在
    os.makedirs(os.path.dirname(output_excel_path), exist_ok=True)
    df.to_excel(output_excel_path, index=False)

    logger.info(f"\n✅ 全部生成完成，共 {len(results)} 条。")
    logger.info(
        f"⚙️ 总推理耗时: {end - start:.2f} 秒，平均每条 {(end - start) / len(results):.2f} 秒。"
    )

if __name__ == "__main__":
```

## 概述

`cached_tokens` 是 vLLM 在 OpenAI 兼容 API 中返回的一个关键指标，用于追踪**前缀缓存命中的令牌数量**。当启用前缀缓存功能时，模型可以复用之前计算过的 KV 缓存，从而加速推理。

在 OpenAI API 响应中，`cached_tokens` 作为 `usage.prompt_tokens_details.cached_tokens` 返回。

### 示例响应

```json
{
  "id": "chatcmpl-83048725323a297d",
  "choices": [...],
  "usage": {
    "prompt_tokens": 25,
    "completion_tokens": 15,
    "total_tokens": 40,
    "prompt_tokens_details": {
      "cached_tokens": 16
    }
  }
}
```

## 核心概念

### 什么是前缀缓存？

前缀缓存（Prefix Caching）是一种优化技术，当多个请求共享相同的前缀时，模型可以：

1. **本地缓存命中**（Local Cache Hit）：在同一 GPU 上复用前面计算过的 KV 缓存
2. **外部 KV 转移**（External KV Transfer）：从其他 GPU/节点转移预计算的 KV 缓存

### cached_tokens 的含义

```
cached_tokens = num_local_cached_tokens + num_external_cached_tokens
```

**例如**：如果一个 25 token 的请求中有 16 个 token 被缓存：
- 实际计算的 tokens：25 - 16 = 9 个
- 缓存的 tokens：16 个
- 加速比：16/25 = 64%

### 3. 内部统计结构

```python
# vllm/v1/metrics/stats.py

@dataclass
class PrefillStats:
    """一次预填充计算的分解统计"""
    num_prompt_tokens: int = 0           # 总的提示令牌数
    num_computed_tokens: int = 0          # 实际计算的令牌数
    num_cached_tokens: int = 0            # 缓存的令牌数 ← 关键
    num_local_cached_tokens: int = 0      # 本地缓存命中
    num_external_cached_tokens: int = 0   # 外部 KV 转移


@dataclass
class PromptTokenStats:
    """按来源分类的提示令牌统计"""
    computed: int = 0                  # 本地计算
    local_cache_hit: int = 0           # 本地缓存命中
    external_kv_transfer: int = 0      # 外部 KV 转移
    cached_tokens: int = 0             # 总缓存 = local_cache_hit + external_kv_transfer
    total: int = 0                     # 总数
    
    def update_from_output(self, prefill_stats: PrefillStats) -> None:
        """从预填充统计更新"""
        self.computed += prefill_stats.num_computed_tokens
        self.cached_tokens += prefill_stats.num_cached_tokens
        self.total += prefill_stats.num_prompt_tokens
        self.local_cache_hit += prefill_stats.num_local_cached_tokens
        self.external_kv_transfer += prefill_stats.num_external_cached_tokens
```

## 常见问题

### Q1: 为什么我的响应中没有 `prompt_tokens_details`？

**答**：有以下几个可能的原因：

1. **未启用详情功能**
   ```bash
   # 需要添加此标志
   vllm serve model --enable-prompt-tokens-details
   ```

2. **前缀缓存未启用**
   ```bash
   # 需要同时启用前缀缓存
   vllm serve model --enable-prefix-caching --enable-prompt-tokens-details
   ```

3. **没有缓存命中**
   - 如果 `num_cached_tokens = 0`，则不会返回详情
   - 确保请求包含可被缓存的公共前缀

### Q2: cached_tokens 应该与 prompt_tokens 相等吗？

**答**：通常不会。`cached_tokens` 是 `prompt_tokens` 中被复用的部分：

```
cached_tokens ≤ prompt_tokens

例如：
- prompt_tokens = 25
- cached_tokens = 16  ← 其中 16 个来自缓存
- computed_tokens = 9 ← 其中 9 个新计算
```

### Q3: 如何优化缓存命中率？

**答**：几个关键策略：

1. **固定系统提示**
   ```python
   # 相同的系统提示使缓存更容易命中
   messages = [
       {"role": "system", "content": "You are a helpful assistant..."},
       {"role": "user", "content": "User question"}
   ]
   ```

2. **分批处理相似请求**
   - 相同的用户场景会共享同样的前缀

3. **增加缓存大小**
   ```bash
   vllm serve model --gpu-memory-utilization 0.9
   ```

4. **使用块大小优化**
   ```bash
   vllm serve model --kv-cache-dtype fp8
   ```

### Q4: cached_tokens 与性能的关系？

**答**：缓存的令牌越多，性能提升越明显：

```
性能收益 ≈ (cached_tokens / prompt_tokens) × 计算成本
```

**实际测试结果**：
- 64% 缓存命中率 → ~40% 延迟降低
- 90% 缓存命中率 → ~70% 延迟降低

### Q5: 在不同的模型上，cached_tokens 如何工作？

**答**：支持情况取决于模型架构：

| 模型类型 | 前缀缓存支持 | 说明 |
|---------|-----------|------|
| Transformer | ✅ | 完全支持 |
| Mamba | ✅ | 需要 `SupportsMambaPrefixCaching` |
| 混合架构 | ⚠️ | 部分支持，可能有限制 |
| MoE | ✅ | 支持，但需配置 |

## 概览（结论先行）

- `cached_tokens`（API 中看到的缓存令牌数） = `num_local_cached_tokens + num_external_cached_tokens`。  
- Cache 命中是基于“块（block）”级别的匹配（按 `block_size` / `hash_block_size` 分块并哈希比较）。  
- 调度器在 prefill 阶段调用 `KVCacheManager` 来查找本地/远端命中，并以 `PrefillStats` 记录结果。  
- 最终统计（`num_cached_tokens`）通过 `RequestOutput` / `Usage`（API 层）返回给客户端（当 `enable_prompt_tokens_details=True` 时）。  
- 关键函数/文件：
  - `vllm/v1/core/kv_cache_manager.py`（`KVCacheManager.get_computed_blocks` / `allocate_slots` / `cache_blocks` 等）  
  - `vllm/v1/metrics/stats.py`（`PrefillStats` / `PromptTokenStats`）  
  - `vllm/outputs.py`（`RequestOutput.num_cached_tokens`）  
  - `vllm/entrypoints/.../serving.py`（API 层把 `num_cached_tokens` 塞到 `Usage` 中）  
  - 协调器/连接器（coordinator / connector）负责具体的本地块查找与外部检索

## 关键数据结构（来自源码）

- `PrefillStats`（`vllm/v1/metrics/stats.py`）
```python
@dataclass
class PrefillStats:
    num_prompt_tokens: int = 0
    num_computed_tokens: int = 0
    num_cached_tokens: int = 0
    num_local_cached_tokens: int = 0
    num_external_cached_tokens: int = 0

    def set(
        self,
        num_prompt_tokens: int,
        num_local_cached_tokens: int,
        num_external_cached_tokens: int,
    ):
        num_cached_tokens = num_local_cached_tokens + num_external_cached_tokens
        assert num_cached_tokens <= num_prompt_tokens

        self.num_prompt_tokens = num_prompt_tokens
        self.num_computed_tokens = num_prompt_tokens - num_cached_tokens
        self.num_cached_tokens = num_cached_tokens
        self.num_local_cached_tokens = num_local_cached_tokens
        self.num_external_cached_tokens = num_external_cached_tokens
```

## 高层执行流程（一步步）

1. 客户端请求到达，`Scheduler` 准备好请求数据（包含 token ids / token blocks / block_hashes 等）。  
   - `Scheduler` 会把请求分成按 block 的哈希（`request.block_hashes`）作为查找输入（block_hashes 的构造在 scheduler/输入预处理里完成）。

2. 在 prefill 阶段，`KVCacheManager.get_computed_blocks(request)` 被调用，目的是“找出已经计算（可以命中）的块”：  
```python
def get_computed_blocks(self, request: Request) -> tuple[KVCacheBlocks, int]:
    if not self.enable_caching or request.skip_reading_prefix_cache:
        return self.empty_kv_cache_blocks, 0

    # NOTE: When all tokens hit the cache, we must recompute the last token
    # to obtain logits. Thus, set max_cache_hit_length to prompt_length - 1.
    max_cache_hit_length = request.num_tokens - 1
    computed_blocks, num_new_computed_tokens = (
        self.coordinator.find_longest_cache_hit(
            request.block_hashes, max_cache_hit_length
        )
    )

    if self.log_stats:
        self.prefix_cache_stats.record(
            num_tokens=request.num_tokens,
            num_hits=num_new_computed_tokens,
            preempted=request.num_preemptions > 0,
        )

    return self.create_kv_cache_blocks(computed_blocks), num_new_computed_tokens
```
   - 说明：通过 `coordinator.find_longest_cache_hit(request.block_hashes, max_cache_hit_length)` 获取本地可重用的块（`computed_blocks`）与命中 token 数（`num_new_computed_tokens`）。  
   - 注意：当全部命中时，会强制保持最后一个 token 作为需重算（`max_cache_hit_length = prompt_len - 1`），原因是需要 logits。

3. 统计本地命中后，调度器可能会尝试外部检索（connector / KV transfer）得到 `num_external_computed_tokens`（这个由外部 connector 负责），并把这些计入 `PrefillStats`。

4. 在分配 slots（`allocate_slots`）时，`KVCacheManager` 会把本地/外部的命中合并计入 `total_computed_tokens`，并在最后调用 `coordinator.cache_blocks(request, num_tokens_to_cache)` 将可缓存的块真正标记为已缓存：
```python
# allocate_slots 中的关键片段（已截取）
num_local_computed_tokens = request.num_computed_tokens + num_new_computed_tokens
total_computed_tokens = min(
    num_local_computed_tokens + num_external_computed_tokens,
    self.max_model_len,
)
...
# 在合适时机，把外部/本地计算的已命中块加入 request 的分配
self.coordinator.allocate_new_computed_blocks(
    request_id=request.request_id,
    new_computed_blocks=new_computed_block_list,
    num_local_computed_tokens=num_local_computed_tokens,
    num_external_computed_tokens=num_external_computed_tokens,
)
...
# 计算要最终缓存的 token 数（排除 draft/未最终确认 token）
num_tokens_to_cache = min(
    total_computed_tokens + num_new_tokens,
    request.num_tokens,
)
self.coordinator.cache_blocks(request, num_tokens_to_cache)
```
   - 说明：
     - `num_external_computed_tokens` 表示外部 connector（如 lmcache）已经缓存的 token 数量。  
     - `num_tokens_to_cache` 是最终要提交到 block pool 的 token 数（会受 sliding-window、block 对齐及 draft token 限制影响）。

5. `PrefillStats` 被设置并传递到上层（engine core 的输出里），示例来源（`vllm/v1/metrics/stats.py`）：
```python
prefill_stats.set(
    num_prompt_tokens=..., 
    num_local_cached_tokens=..., 
    num_external_cached_tokens=...
)
# 结果： prefill_stats.num_cached_tokens = local + external
```

6. API 层（在构建最终响应时）会把 `num_cached_tokens`（来自最终 `RequestOutput` 或 engine 输出）写入 `Usage.prompt_tokens_details`（只有在 `enable_prompt_tokens_details=True` 时）：
```python
# 伪示意（来自 vllm/entrypoints/.../serving.py）
if self.enable_prompt_tokens_details and num_cached_tokens:
    final_usage.prompt_tokens_details = PromptTokenUsageInfo(cached_tokens=num_cached_tokens)
```

## 重要实现细节与设计权衡（源码可见）

- 块级别对齐（`block_size` / `hash_block_size`）  
  - 查询与缓存是基于 block 的；`block_size` 决定了命中粒度。部分块命中会受对齐规则影响（可能需要重算或整块分配）。  
  - 当 `block_size > hash_block_size` 时，会先按 `hash_block_size` 计算，再合并为 `block_size`（见 `KVCacheManager` 初始化时传入的 `scheduler_block_size` / `hash_block_size`）。

- 全命中时必须重算最后一个 token  
  - 为了获得 logits（用于输出 logprobs 等），若前缀完全命中，系统会重新计算最后一个 token（因此 `get_computed_blocks` 使用 `max_cache_hit_length = prompt_len - 1`）。

- 外部 KV（connector）与异步加载  
  - 外部命中（external）会被计入 `num_external_computed_tokens`。外部命中可能需要异步传输到本地并在后续步骤分配与缓存（`allocate_slots` 支持 `num_external_computed_tokens` 和 `delay_cache_blocks`）。  
  - 当使用外部 KV 时，系统会在 `allocate_slots` 里调用 `allocate_new_computed_blocks()` 把这些外部块临时视为“已计算”，再由 `coordinator.cache_blocks` 将它们标记/加入本地 block pool。

- sliding window / eviction / watermark  
  - `KVCacheManager` 会考虑 sliding-window（只保留可复用范围内的 KV）和 watermark（保留最小空闲块以减少频繁抢占）。这些影响是否能把新请求放入缓存、是否会触发回收。

- 预取/投机（speculative）与 draft token  
  - `allocate_slots` 支持 `num_lookahead_tokens`（投机 token），但只有“经确认”的 token 才会被真正缓存（函数里有 cap 到 `request.num_tokens` 以排除未确认 draft token 的缓存）。

## 从源码追踪到 API 的路径（简要）

1. `Scheduler` 触发 prefill → 调用 `KVCacheManager.get_computed_blocks(...)`（`vllm/v1/core/kv_cache_manager.py`）  
2. `KVCacheManager` 与 coordinator 协作找本地命中（`coordinator.find_longest_cache_hit`）并记录 prefix cache stats（若 `log_stats=True`）  
3. 调度/connector 再合并外部命中（若有），然后 `allocate_slots` 分配并最终调用 `cache_blocks` 将块标记为缓存  
4. EngineCore 输出包含 `prefill_stats`（`PrefillStats`），最终会被封装成 `RequestOutput.num_cached_tokens`（`vllm/outputs.py`）  
5. API 层（openai-serving / serve）读取 `RequestOutput.num_cached_tokens`，并在 `enable_prompt_tokens_details=True` 时填入 `Usage.prompt_tokens_details.cached_tokens`（`vllm/entrypoints/.../serving.py`）

## 常见坑 / 注意事项（基于源码与实现）

- Block 对齐导致的“最后一个块必须重算”会使看上去实际计算的 token 数多于预期（即 `num_computed_tokens` 可能不是简单的 `prompt_tokens - cached_tokens`）。  
- 当请求使用 pooling / prompt_logprobs / 特殊功能时，代码可能会跳过 prefix cache 读（`skip_reading_prefix_cache`），导致 `cached_tokens` 为 0。  
- 外部 connector 的加载延迟或失败会影响最终的 `num_external_cached_tokens` 的计数（系统设计允许异步加载与延迟提交）。  
- 在 high-parallel / 大 GPU 上，配合 speculative decoding 可能会产生与 block 表形状相关的 bug（需要注意 block 数与 speculative block 数的配套）。  
- `cache_blocks` 会根据 sliding window 清理过时的块；因此缓存命中不是永久保持的，需要考虑工作负载的重用模式。

## 调优建议（实践）

- 保证前缀稳定：尽量把固定的 system prompt / 导航模板放在前缀，减少随机/时间戳等变化（提高命中率）。  
- 选择合适 `block_size` / `hash_block_size`：若你的 prompt 模式是重复长短块，调小 `block_size` 可以提高命中细粒度；但会增加 metadata 与哈希开销。权衡后选择最合适值。  
- 增大可用缓存（GPU 内存分配）：通过 `--gpu-memory-utilization` 增加 block 数量或调整 kv cache 参数，提升整体缓存容量与命中率。  
- 使用外部 KV（lmcache / mooncake）在多节点部署下能极大提高跨节点重用，但要监控网络/传输延迟。  
- 如果使用 speculative decode（MTP 等），务必测试与 prefix caching 的组合（历史上有交互 bug），并留意 decode/startup 时的 cudagraph/shape 错误。
